GitHub Actions workflows

GitHub Actions workflows

Short overview of the workflows in this repository.

Linting specification

Runs different validations and lintings on the source specification.

Purpose
Ensures OEAPI remains technically correct and consistent with the OpenAPI specification.

Lint OpenAPI CLI

Runs OpenAPI validation and linting on the source specification.

Workflow: lint-source-openapi.yaml

Value
Config.openapi/lint.yaml
Local installgo install github.com/speakeasy-api/openapi/cmd/openapi@latest
Local commandopenapi spec lint source/spec.yaml -c .openapi/lint.yaml

Lint Redocly

Runs Redocly validation rules on the specification.

Workflow: lint-source-redocly.yml

Value
Configredocly.yaml
Custom rulesredocly-plugin.cjs
Local installnpm install -g @redocly/cli
Local commandredocly lint source/spec.yaml

Lint Vacuum (not active yet)

Runs Vacuum validation rules on the specification. This check is currently not implemented as a workflow.

Workflow: lint-source-vacuum.yaml

Value
Configvacuum.conf.yaml
.spectral.yaml
Local installnpm install -g @quobix/vacuum
Local commandvacuum lint source/spec.yaml

Blocking PRs

Block PRs from release/5.*

Blocks pull requests that originate from release/5.* branches.

Workflow: block-prs-from-release-5.yaml

Purpose
Prevents changes from legacy release branches being merged into supported branches.

Update files

Auto re-approve when only CHANGELOG changed

Automatically re-approves a pull request when the only change after an approval is an update to CHANGELOG.md.

Workflow: auto-reapprove-changelog.yaml

Purpose
Avoids unnecessary manual re-approval when the changelog is updated after a review.

Check changelog

Checks whether CHANGELOG.md is updated in a pull request.

Workflow: check-changelog.yaml

Purpose
Ensures every functional change is recorded in the changelog.

Bypass team
This check may be bypassed because it verifies documentation discipline rather than technical correctness. In rare situations a pull request may need to be merged while the changelog is updated later or the change does not require a changelog update.

Publish and release

Create and publish new release from tag

Builds and publishes a release when a version tag is created. If it does not exist, the workflow also creates the corresponding release branch.

Workflow: release-from-tag.yaml

Purpose
Creates the official released version of the specification and publishes it to the documentation site and release artefacts.

Publish Zudoku specification by hand from a release branch

Manually republishes Zudoku documentation from a release branch.

Workflow: republish_documentation.yaml

Purpose
Allows controlled republishing when something needs to be rebuilt without creating a new release. Needed for example for documentation changes in the code.

Create and publish from main (unreleased)

Publishes the documentation from main to the unreleased section of the documentation site.

Workflow: unreleased-docs.yaml

Purpose
Keeps the public preview documentation in sync with the current development state.
